# Create a Model Response

<Tip title="💡 Troubleshooting">
Encountering errors? See the [API Error Codes Reference](/en/guide/advanced/error-codes) for a complete list of error types and troubleshooting steps.
</Tip>

```
POST https://api.agipower.ai/v1/responses
```

The Create a Model Response endpoint is compatible with OpenAI’s [Create a Model Response](https://platform.openai.com/docs/api-reference/responses/create?lang=curl) endpoint. It is used to create a single model response (supports text/image/file inputs; text or structured JSON outputs; built-in tools and function calling, etc.).

Below is a list of all parameters that may be supported by different models. Parameter support varies by model; for the exact parameters supported by each model, see the model’s detail page.

## Request headers

### Authorization `string` **Required**

Bearer token authentication.

### Content-Type `string` **Required**

Request content type. Default is `application/json`.

## Request body

### model `string` **Required**

The model ID for this inference call, in the format `&lt;provider&gt;/&lt;model_name&gt;`, e.g., openai/gpt-5. You can find it on each model’s detail page.

### input `string | array` **Optional**

The content passed to the model.

#### `string`

- Treated as plain-text input (equivalent to a single `user` text input).

#### `array` (Input item list)

Each element in the array is an **Item** (a context item). It can include messages, images, files, tool outputs, references, etc. Possible types include:

##### 1. Input message `object`

- type `string` **Optional**: fixed to `message`
- role `string` **Required**: `user` / `assistant` / `system` / `developer`
- content `string | array` **Required**: message content
  - If `string`: plain text
  - If `array`: a list of content parts (see Input content parts below)

#### Input content parts (possible input content part types)

1. Input text `object`

- type `string` **Required**: `input_text`
- text `string` **Required**: text content

2. Input image `object`

- type `string`: `input_image` **Required**
- detail `string`: `high` / `low` / `auto` **Required**: image processing detail level; affects token usage and recognition accuracy
- file_id `string` **Optional**: ID of an uploaded file
- image_url `string` (URL or data URL/base64) **Optional**: image URL or base64-encoded data

3. Input file `object`

- type `string`: `input_file` **Required**
- file_data `string` **Optional**: base64-encoded file data
- file_id `string`**Optional**: ID of an uploaded file
- file_url `string`**Optional**: file URL
- filename `string`**Optional**: file name



##### 2. Item types overview

#### Input message `object`

- type `string` **Optional**: fixed to `message`
- role `string` **Required**: `user` / `assistant` / `system` / `developer`
- status `string` **Optional**: `in_progress` / `completed` / `incomplete` item status
- content `string | array` **Required**: message content
  - If `string`: plain text
  - If `array`: a list of content parts; each element can be `input_text` / `input_image` / `input_file` (same as above)



#### Output message (model output message) `object`

(Used when you “feed back the model’s previous output message item” into `input`.)

- type `string`**Required**: `message`
- role `string`**Required**: fixed to `assistant`
- id `string`**Required**: unique ID of the output message
- status `string`**Required**: `in_progress` / `completed` / `incomplete`
- content `array`**Required**: list of **Output content parts** (see below)

###### Output content parts (possible output content part types)

- type `string`**Required**: `output_text`
- text `string`**Required**: output text
- annotations `array`**Required**: annotations (see below)
- logprobs `array`**Required**: logprobs (see below)

**annotations** element types:

- File citation `object`
  - type `string`**Required**: `file_citation`
  - file_id `string`**Required**: cited file ID
  - filename `string`**Required**: cited file name
  - index `integer`**Required**: starting index position in the text
- URL citation `object`
  - type `string`**Required**: `url_citation`
  - start_index `integer`**Required**: starting index position of the citation in the text
  - end_index `integer`**Required**: ending index position of the citation in the text
  - title `string`**Required**: cited page title
  - url `string`**Required**: cited page URL
- Container file citation `object`
  - type `string`**Required**: `container_file_citation`
  - container_id `string`**Required**: container file ID
  - file_id `string`**Required**: actual cited file ID
  - filename `string`**Required**: cited file name
  - start_index `integer`**Required**: starting index position of the citation in the text
  - end_index `integer`**Required**: ending index position of the citation in the text
- File path `object`
  - type `string`**Required**: `file_path`
  - file_id `string`**Required**: file ID
  - index `integer`**Required**: index position in the list

**logprobs** (each element is logprob info for one token; field hierarchy below):

- bytes `array`**Required**: byte representation of the token
- logprob `number`**Required**: token log probability
- token `string`**Required**: token text
- top_logprobs `array`**Required**: most likely candidate tokens (each element includes `bytes` / `logprob` / `token`)

2. Refusal `object`

- type `string`**Required**: `refusal`
- refusal `string`**Required**: explanation for why the model refused to answer



#### Function tool call `object`

The model initiates a call to a **function tool**.

- type `string`: `function_call` **Required**
- call_id `string` **Required**: unique identifier for the function call
- name `string` **Required**: name of the function to call
- arguments `string` **Required**: function arguments as a JSON string
- id `string` **Optional**: unique item ID
- status `string` **Optional**: call status, `in_progress` / `completed` / `incomplete`



#### Function tool call output (function tool output) `object`

Send the result of executing the function back to the model (for the next step of reasoning).

- type `string`: `function_call_output` **Required**
- call_id `string`**Required**: corresponding function call identifier
- output `string | array` **Required**: function execution result, either a string or a list of content parts
- id `string` **Optional**: unique output item ID
- status `string` **Optional**: output status, `in_progress` / `completed` / `incomplete`

> When `output` is an `array`, elements can be `input_text` / `input_image` / `input_file` (same as above).



#### Custom tool call `object`

The model initiates a call to a “custom tool” (you must execute it on your side).

- type `string`: `custom_tool_call`**Required**
- call_id `string`**Required**: unique identifier for the custom tool call
- name `string`**Required**: name of the custom tool to call
- input `string`**Required**: input passed to the tool
- id `string` **Optional**: unique item ID



#### Custom tool call output (custom tool output) `object`

Send the execution result of the custom tool back to the model.

- type `string`: `custom_tool_call_output`**Required**
- call_id `string`**Required**: corresponding custom tool call identifier
- output `string | array`**Required**: tool result, either a string or a list of content parts
- id `string`**Optional**: unique output item ID

> When `output` is an `array`, elements can be `input_text` / `input_image` / `input_file` (same as above).



#### Web search tool call `object`

- type `string`**Required**: `web_search_call`
- id `string`**Required**: unique ID for the web search tool call
- status `string`**Required**
- action `object`**Required**: describes what the web tool did in this call (search/open_page/find)

###### Possible action types

1. Search action `object`

- type `string`**Required**: action type, fixed to `search`
- query `string`**Required**: search query (deprecated)
- queries `array`**Optional**: list of search queries
- sources `array`**Optional**: list of sources
  - each source:
    - type `string`**Required**: `url`
    - url `string`**Required**: source URL

2. Open page action `object`

- type `string`**Required**: action type, fixed to `open_page`
- url `string`**Required**: page URL to open

3. Find action `object`

- type `string`**Required**: action type, fixed to `find`
- url `string`**Required**: page URL to search
- pattern `string`**Required**: search pattern or keyword




#### Computer tool call (Computer Use call) `object`

The model initiates a Computer Use action (you must execute it on your side). After execution, send results such as screenshots back via `computer_call_output`.

- type `string`**Required**: `computer_call`
- call_id `string` **Required**: unique identifier for the computer use call
- action `object` **Required**: parameters for the specific computer action (see Action types below)
- pending_safety_checks `array` **Optional**: list of safety checks pending user confirmation
  - id `string` **Optional**: unique safety check identifier
  - code `string` **Optional**: safety check code
  - message `string` **Optional**: safety check message
- id `string` **Optional**: unique item identifier
- status `string` **Optional**: call status, `in_progress` (in progress) / `completed` (completed) / `incomplete` (incomplete)

###### Action types (possible `action` variants)

###### Click `object`

- type `string`: `click` **Required**: action type, fixed to `click`
- x `number` **Required**: click X coordinate (pixels)
- y `number` **Required**: click Y coordinate (pixels)
- button `string` **Optional**: mouse button, `left` / `right` / `middle`

###### Double click `object`

- type `string`: `double_click` **Required**: action type, fixed to `double_click`
- x `integer` **Required**: double-click X coordinate (pixels)
- y `integer` **Required**: double-click Y coordinate (pixels)

###### Move `object`

- type `string`: `move` **Required**: action type, fixed to `move`
- x `integer` **Required**: target X coordinate (pixels)
- y `integer` **Required**: target Y coordinate (pixels)

###### Screenshot `object`

- type `screenshot`：`screenshot` **Required**: action type, fixed to `screenshot`

###### Drag `object`

- type `string`: `drag` **Required**: action type, fixed to `drag`
- path `array` **Required**: list of coordinates for the drag path
  - x `integer` **Required**: X coordinate (pixels)
  - y `integer` **Required**: Y coordinate (pixels)

###### Scroll `object`

- type `string`: `scroll` **Required**: action type, fixed to `scroll`
- x `integer` **Optional**: scroll anchor X (pixels; provided by some implementations)
- y `integer` **Optional**: scroll anchor Y (pixels)
- scroll_x `integer` **Optional**: horizontal scroll distance (pixels)
- scroll_y `integer` **Optional**: vertical scroll distance (pixels)

###### Type `object`

- type `string`: `type` **Required**: action type, fixed to `type`
- text `string` **Required**: text to type

###### Keypress `object`

- type `string`: `keypress` **Required**: action type, fixed to `keypress`
- keys `array` **Required**: key combo to press, e.g. `["CTRL","L"]` means Ctrl+L

###### Wait `object`

- type `string`: `wait` **Required**: action type, fixed to `wait`



#### Computer tool call output (Computer Use output) `object`

- type `string`**Required**: `computer_call_output`
- call_id `string`**Required**: corresponding computer use call identifier
- output `object`: output of the computer action (e.g., screenshots)

#### Shell tool call `object`

##### 3. Item reference `object`

Used to reference an existing item (internal identifier).

- type `string`**Required**: `item_reference`
- id `string`**Optional**

### instructions `string` **Optional**

System/developer instructions inserted into the context. When used with `previous_response_id`, it will not inherit the previous round’s instructions, making it easier to replace the system prompt.

### previous_response_id `string` **Optional**

ID of the previous Response for continuing multi-turn conversations; **cannot be used together with `conversation`**.

### conversation `string | object` **Optional** (default `null`)

Associates this response with a conversation. The conversation’s items will be prepended to this request’s input context, and after completion this request’s input/output will be automatically appended to the conversation.

- Conversation ID `string`: unique conversation ID used to identify and reuse conversation history
- Conversation object `object`
  - id `string`**Required**: unique conversation ID for persistence and retrieval

### prompt `object` **Optional**

Reference a prompt template and pass variables.

- id `string` **Required**: prompt template ID
- variables `object`**Optional**: template variable key-value pairs for placeholder substitution
- version `string`**Optional**: prompt template version

### include `array` **Optional**

Specifies additional output data to include in the response. Supported values include (string enum):

- `web_search_call.action.sources`: include source info for web search calls
- `computer_call_output.output.image_url`: include screenshot URL from Computer Use tool outputs
- `message.input_image.image_url`: include image URLs in input messages
- `message.output_text.logprobs`: include per-token log probabilities for output text
- `reasoning.encrypted_content`: include encrypted reasoning content

### max_output_tokens `integer` **Optional**

Caps the maximum tokens the response can generate (includes visible output tokens and reasoning tokens).

### max_tool_calls `integer` **Optional**

Caps the total number of **built-in tool** calls that can be processed within this response (accumulated across tools).

### parallel_tool_calls `boolean` **Optional** (default `true`)

Whether to allow parallel tool calls.

### reasoning `object` **Optional** (reasoning models only)

Reasoning model configuration.

- effort `string` **Optional** (default `medium`)  
  Controls reasoning intensity, affecting reasoning depth and compute time. Supported: `none` / `minimal` / `low` / `medium` / `high` / `xhigh`
- summary `string` **Optional**  
  Reasoning summary level: `auto` / `concise` / `detailed`
- generate_summary `string` (Deprecated)  
  Deprecated: use `summary` instead

### text `object` **Optional**

Text output configuration (including structured output formats).

- format `object` **Optional**: output format (default `\{ "type": "text" \}`)  
  Possible types:
  1. Text `object`
     - type `string`**Required**: `text`: plain text output

  2. JSON schema `object` (Structured Outputs)
     - type `string`**Required**: `json_schema`: structured JSON output
     - name `string`**Required**: output format name identifier
     - schema `object`**Required**: JSON Schema defining the output structure
     - description `string` **Optional**: description of the output format
     - strict `boolean` **Optional** (default `false`): whether to strictly enforce the schema; when true, validation is enforced

  3. JSON object `object` (legacy JSON mode)
     - type `string`**Required**: `json_object`: JSON object output

- verbosity `string` **Optional** (default `medium`)  
  Constrains response verbosity. Lower values produce more concise responses; higher values produce more detailed responses. Supported values: `low` / `medium` / `high`

### temperature `number` **Optional** (default `1`)

Sampling temperature, typically 0–2. Prefer tuning either this or `top_p`, not both.

### top_p `number` **Optional** (default `1`)

Nucleus sampling parameter.

### top_logprobs `integer` **Optional**

0–20. Number of most likely tokens to return at each position.

### truncation `string` **Optional** (default `disabled`)

Context truncation strategy for handling prompts that exceed the model’s context window:

- `auto`: when content exceeds the context window, automatically drop the earliest items from the start of the conversation to make room
- `disabled`: disables auto-truncation; requests that exceed the context window will fail (400)

### tools `array` **Optional**

Declares the list of tools the model is allowed to call in this request. The model may choose to call these tools as needed to enhance its capabilities.

#### Possible `tools[i]` types

Below are the supported tool type definitions. Each tool type has specific configuration parameters:

#### Function tool (function tool definition) `object`

- type `string`**Required**: `function`: tool type, fixed to function tool
- name `string`**Required**: unique function name identifier
- parameters `object`**Required**: JSON Schema defining function parameters (structure + validation rules)
- strict `boolean`**Required** (default `true`): whether to strictly validate parameters against the schema
- description `string` **Optional**: detailed function description to help the model decide when to use it




#### Web search (generally available) `object`

- type `string` **Required**: `web_search` or `web_search_2025_08_26`
- filters `object` **Optional**: result filters
  - allowed_domains `array[string]` **Optional** (default `[]`): allowed domain allowlist (subdomains allowed)
- search_context_size `string` **Optional** (default `medium`): search context size level
- user_location `object` **Optional**: approximate user location for localized results
  - type `string` **Required**: `approximate`
  - city `string` **Optional**
  - country `string` **Optional**: 2-letter ISO (e.g. `US`)
  - region `string` **Optional**
  - timezone `string` **Optional**: IANA TZ (e.g. `America/Los_Angeles`)



#### Web search preview `object`

- type `string`**Required**: `web_search_preview` or `web_search_preview_2025_03_11`
- search_context_size `string` **Optional**: `low` / `medium` / `high`
- user_location `object` **Optional**: user location
  - type `string`**Required**: `approximate`
  - city `string` **Optional**
  - country `string` **Optional** (2-letter ISO)
  - region `string` **Optional**
  - timezone `string` **Optional** (IANA TZ)



#### Computer use preview (Computer Use tool) `object`

- type `string` **Required**: `computer_use_preview`
- display_width `integer` **Required**: display width (pixels)
- display_height `integer` **Required**: display height (pixels)
- environment `string` **Required**: environment type (e.g. `browser` / `mac` / `windows` / `ubuntu`)




#### Apply patch tool (apply patch) `object`

- type `string`**Required**: `apply_patch`



#### MCP tool (remote MCP tool) `object` (if enabled)

- type `string` **Required**: `mcp`
- server_label `string` **Required**: MCP server label (for tool-call routing)
- server_url `string` **Optional**: remote MCP server
- authorization `string` **Optional**: OAuth access token (for remote MCP server or service connector)
- allowed_tools `array[string] | object` **Optional**: allowed tools list or filter
  - **MCP allowed tools** `array[string]`
  - **MCP tool filter** `object`
    - read_only `boolean` **Optional**: filter by read-only tools
    - tool_names `array[string]` **Optional**: tool name allowlist
- require_approval `string | object` **Optional**: which tools require approval (example uses `never`; supports finer granularity)
  - `string`: `always` / `never`
  - `object`: approval filter (schema in docs)




#### Shell tool (generic Shell) `object`

- type `string` **Required**: `shell`

(Usually no extra configuration is needed; execution parameters are in the model output’s `shell_call.action`.)



#### Local shell tool (legacy local Shell) `object`

    - type `string` **Required**: `text`
  - **Grammar format** `object`
    - type `string` **Required**: `grammar`
    - syntax `string` **Required**: `lark` / `regex`
    - definition `string` **Required**: grammar definition

#### MCP tool (force a specific MCP tool) `object`

- type `string`**Required**: `mcp`

#### Custom tool (force a specific custom tool) `object`

- type `string`**Required**: `shell`




### store `boolean` **Optional** (default `true`)

Whether to store this Response for later retrieval.

### provider `object` **Optional**

Used to configure routing and failover strategies across multiple model providers (e.g., OpenAI, Anthropic, Google) for this request.
If not configured, the project or model’s default routing strategy is used.

#### routing `object` **Required**

Routing strategy configuration that determines how requests are selected and distributed across multiple providers.

##### type `string` **Required**

Routing type. Supported values:

- `priority`
  Select providers by priority order: try the first one, then try the next upon failure (can be combined with fallback).
- `round_robin`
  Round-robin distribution: evenly distribute traffic across providers.
- `least_latency`
  Lowest-latency first: choose the provider with the fastest response based on historical/real-time stats.

##### primary_factor `string` **Optional**

Primary factor when multiple providers are available. For example:

- `cost`
  Prefer lower-cost providers
- `speed`
  Prefer faster providers
- `quality`
  Prefer higher-quality providers (e.g., stronger models / more stable)

Actual behavior combines with `type`. For example, when `type = "priority"`, `primary_factor` mainly affects how priorities are ordered.

##### providers `array` **Required**

List of model providers that can participate in routing. Example: `["openai", "anthropic", "google"]`

#### fallback `string` **Optional**

Failover strategy. When the currently selected provider fails (e.g., timeout, insufficient quota, service unavailable), how to switch automatically:

`"true"`: enable automatic failover. When the current provider is unavailable, try other available providers according to the routing strategy.

`"false"`: disable failover. If the current provider call fails, return the error directly without trying other providers.

`"&lt;provider_name&gt;"`: explicitly specify a fixed backup provider, e.g. `"anthropic"`:

Use the provider selected by primary routing first  
If it fails, switch to the specified fallback provider  
If both primary and fallback fail, return an error

### model_routing_config `object` **Optional**

Used to configure selection and routing among different models **within the same provider** for the current request (e.g., choosing between `gpt-4o`, `gpt-4-turbo`, `claude-3-5-sonnet`).

If not configured, the project or SDK’s default model selection strategy is used (e.g., default model, default task-type mapping).

#### available_models `array` **Required**

List of **model names** that can be used for routing or fallback.

#### preference `string` **Optional**

Preferred model name.

#### task_info `object` **Optional**

Task metadata used to determine the model or parameters **based on task type and complexity**.

Internal fields:

##### task_type `string` **Required**

Task type describing the purpose of the request for routing or automatic parameter selection.

- Example supported values:
  - `"chat"` — conversational tasks (multi-turn chat, assistant Q&A)
  - `"completion"` — general text generation/completion
  - `"embedding"` — vectorization/semantic embeddings
- Usage:
  - Can set different default models or quota strategies per task type
  - Can be combined with `complexity` to decide whether to use a stronger model

##### complexity `string` **Optional**

Task complexity indicating the difficulty or importance of the request.

- Supported values:
  - `"low"` — simple tasks (short answers, simple rewrites, etc.)
  - `"medium"` — medium complexity (general Q&A, basic code, standard analysis)
  - `"high"` — high complexity (long document analysis, complex programming, large-scale reasoning)
- Usage:
  - Select different tiers of models by complexity (e.g., cheaper models for low complexity; stronger models for high complexity)
  - Can also be used to control timeouts, retry strategies, etc.

##### additional_properties `object` **Optional**

Additional task-related fields as an extensible key-value map.

#### additional_properties `object` **Optional**

Additional fields for the model routing config itself, used to attach extra control info beyond the standard structure.

## Response (streaming)

When `stream: true`, `POST /v1/responses` does not return the full Response object in one shot. Instead, it continuously pushes a sequence of **Streaming events** via **SSE (Server-Sent Events)**. ([platform.openai.com](https://platform.openai.com/docs/guides/streaming-responses))

### Transport format (SSE)

- HTTP responses are typically `Content-Type: text/event-stream`
- The server sends multiple events in order
- Each event, at the “wire protocol” level, typically looks like this (example):

```text
data: {"type":"response.output_text.delta","item_id":"msg_123","output_index":0,"content_index":0,"delta":"Hello","sequence_number":12}

```

> Note: SSE also supports an `event: ...` line, but in Responses streaming semantics, the event type is determined by the JSON `type` field. SDKs also parse each event directly into an object for consumption. ([platform.openai.com](https://platform.openai.com/docs/guides/streaming-responses))

### General notes on Streaming events

- Each event includes:
  - `type`: event type string (e.g., `response.created`)
  - `sequence_number`: event sequence number for ordering (examples may start from 0 or 1)
- Many events include indexes/association fields such as `output_index`, `item_id`, `content_index` to precisely merge delta events back into the specific item in `output[]` and the specific content part within that item.

## Streaming events

> Below, each event lists fields in the same order as on the official reference page. The **exact structures** of objects like `response` / `item` / `part` / `annotation` match the earlier “non-streaming Response object / output items” section (or the corresponding official schemas).
